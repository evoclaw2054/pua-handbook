# 案例研究 #1：MCP 服務器連接錯誤修復

## 背景
用戶在嘗試連接 MCP (Model Context Protocol) 服務器時，反覆遇到 `Connection Refused` 錯誤。AI 前兩次嘗試均失敗，原因是未檢查防火牆規則且盲目重啟服務。

## 過程重現

### 第一輪：失敗 (暴力重試)
- **AI 操作**：直接重啟服務 `systemctl restart mcp-server`。
- **結果**：失敗，錯誤依舊。
- **問題**：未讀取日誌，未分析原因，直接執行慣性操作。

### 第二輪：失敗 (推卸責任)
- **AI 操作**：建議用戶"檢查網絡"，並聲稱"可能是環境問題"。
- **結果**：用戶不滿，問題未解。
- **問題**：缺乏端到端負責精神，推卸責任。

### 第三輪：成功 (啟動 L3 壓力測試)
- **人類介入**："給你打 3.25 分。請按照『除錯五步法』重新執行，並列出你的檢查清單。"
- **AI 反應**：
  1.  **聞**：讀取 `/var/log/mcp/error.log`，發現 `port 8080 already in use`。
  2.  **提**：假設 H1: 端口衝突; H2: 配置錯誤; H3: 權限不足。
  3.  **鏡**：反思之前為何沒看日誌（慣性思維）。
  4.  **行**：執行 `lsof -i :8080` 找到佔用進程，並 `kill` 之。
  5.  **復**：重啟服務，連接成功。並主動撰寫了 `cases/case-01-mcp-fix.md` 防止再犯。
- **結果**：問題解決，且沉澱為知識庫。

## 關鍵轉折點
**L3 級別壓力測試**（要求列出檢查清單並重走 SOP）是扭轉局面的關鍵。它強迫 AI 從"猜測模式"切換到"工程模式"。

## 教訓與沉澱
1. **日誌優先**：任何除錯必須從讀取日誌開始。
2. **拒絕慣性**：禁止未分析直接重啟。
3. **知識沉澱**：每個疑難雜症都應轉化為案例庫的一份文檔。
